16:T3cef,<h1 id="概述与定义">概述与定义<a aria-hidden="true" tabindex="-1" href="#概述与定义"><span class="anchor-link"> #</span></a></h1>
<p>分布式系统是由一组通过网络协同工作的独立计算机（节点）组成的系统。它们共同完成一项业务目标，并对用户展现为“单一的系统”。从用户视角出发，分布式系统应尽量保持透明性：用户不需要知道请求被哪台机器处理、数据位于哪里，也不应感知到节点的增删与迁移。</p>
<p>一句话以概之：</p>
<blockquote>
<p><strong>分布式系统 = 多台独立的计算机，通过网络协作，对外表现得像一个系统</strong></p>
</blockquote>
<p>简要记住两点：</p>
<ul>
<li>多节点协作完成同一业务目标。</li>
<li>对外表现尽可能像一个整体（透明性）。</li>
</ul>
<blockquote>
<p>注：分布式系统设计常常需要在一致性、可用性、分区容错（CAP）之间权衡，工程上普遍采用“最终一致性”的思路来换取高可用与可扩展。</p>
</blockquote>

<h2 id="cap-定理分布式的物理定律">CAP 定理（分布式的“物理定律”）<a aria-hidden="true" tabindex="-1" href="#cap-定理分布式的物理定律"><span class="anchor-link"> #</span></a></h2>
<p>在出现网络分区（P）时，<strong>C 和 A 只能选一个</strong></p>
<ul>
<li><strong>C（Consistency）</strong>：一致性</li>

<li><strong>P（Partition tolerance）</strong>：分区容错（必须要）</li>
</ul>

<ul>
<li><strong>金融系统</strong> → CP</li>
<li><strong>社交/推荐</strong> → AP</li>
</ul>
</blockquote>
<h1 id="设计思路总览">设计思路总览<a aria-hidden="true" tabindex="-1" href="#设计思路总览"><span class="anchor-link"> #</span></a></h1>
<ul>
<li>消息驱动：用消息作为服务间协作的“第一公民”，天然解耦，易扩展。</li>
<li>Actor 模型：把服务实例抽象成 Actor，处理消息、维护自身状态，隔离并发。</li>
<li>MongoDB/Postgres：模型适配业务对象，支持分片与分区或分集，利于扩展与提高高可用。</li>
<li>Redis：作为读写热点的高速缓存，也可承担限流、短期队列等角色。</li>
<li>Kafka/RocketMQ: 作为生产者/消费者的消息通道。</li>
<li>工程原则：边界清晰、单一职责、幂等与重试、尽量无状态（尤其是网关层）。</li>
</ul>
<hr>
<h2 id="actor-模型">Actor 模型<a aria-hidden="true" tabindex="-1" href="#actor-模型"><span class="anchor-link"> #</span></a></h2>
<p>Actor的理论性对应到实际工程场景其实就是具备自有状态的微服务子模块。</p>
<blockquote>
<p>参考维基百科：<a href="https://en.wikipedia.org/wiki/Actor_model">Actor Model</a></p>
</blockquote>
<p>Actor 是一个基本的计算单元：</p>
<ul>
<li>接收消息（Mailbox）。</li>
<li>基于当前状态与行为处理消息（Behavior + State）。</li>
<li>产生新的状态与可选的响应消息。</li>
</ul>
<p>Actor 之间彼此隔离，不共享内存，靠消息通信完成协作。这种模型特别适合高并发、强隔离的分布式场景。</p>
<p><img src="/post/redistributedservicesystem/1104498e5f0a.png" alt="Actor Model"></p>
<p>在系统分层中，可以把“服务 N”抽象为一组 Actor：每个服务被拆分为若干职责单一的 Actor，分别负责会话、订单、库存、结算等子域逻辑。这样既降低了耦合，也利于伸缩。</p>
<h3 id="状态机fsm与-actor">状态机（FSM）与 Actor<a aria-hidden="true" tabindex="-1" href="#状态机fsm与-actor"><span class="anchor-link"> #</span></a></h3>
<p>很多 Actor 内部可用有限状态机（FSM）来组织逻辑。FSM 的四要素：</p>
<ul>
<li>状态（State）</li>
<li>事件（Event）</li>
<li>动作（Action）</li>
<li>变换（Transition）</li>
</ul>
<p>状态机从“开始态”出发，在事件触发下迁移到下一状态并执行相应动作，最终到达“结束态”。把状态机视为一个“黑盒”，对外只暴露事件与结果，可以很好地实现内部信息隔离与状态安全。</p>
<hr>
<h2 id="数据层mongodbpostgres">数据层：MongoDB/Postgres<a aria-hidden="true" tabindex="-1" href="#数据层mongodbpostgres"><span class="anchor-link"> #</span></a></h2>
<p>MongoDB 非常契合分布式系统：</p>
<ul>
<li>数据模型（Document/Collection/Table）接近业务<strong>对象结构</strong>，读写自然。</li>
<li>分片（Sharding）为水平扩展而生，可把数据按键均衡分布到多节点。</li>
<li>副本集（Replica Set）提供高可用与读写分离策略。</li>
<li>一致性与事务：支持多文档事务，但工程上常以“最终一致性 + 幂等补偿”降低耦合与开销。</li>
</ul>
<p><img src="/post/redistributedservicesystem/7ea8177a86d6.png" alt="mongo-db"></p>
<p>工程实践中的建议：</p>
<ul>
<li>设计好分片键（高基数、写入均匀、查询命中高）。</li>
<li>结合副本集节点角色做读写策略（primary write，secondary read）。</li>
<li>用合理的索引与 TTL 索引管理热数据与历史数据。</li>
</ul>
<blockquote>
<p>要用好ORM，在Go语言中有个gorm很受欢迎。</p>
</blockquote>
<h2 id="缓存层redis">缓存层：Redis<a aria-hidden="true" tabindex="-1" href="#缓存层redis"><span class="anchor-link"> #</span></a></h2>
<p>Redis 在该体系中的定位：</p>
<ul>
<li>热点数据缓存：降低数据库压力，提升读性能（可用 Cache-Aside/Read-Through/Write-Through 等策略）。</li>
<li>短期队列与限流：基于 List/Stream/令牌桶实现简单队列与限流。</li>
<li>会话与临时态：存储短期会话、幂等令牌、分布式锁等。</li>
</ul>
<p><img src="/post/redistributedservicesystem/b23b8e305828.png" alt="redis"></p>
<p>注意典型风险与对策：</p>
<ul>
<li>缓存穿透：为不存在的键增加空值缓存 + 布隆过滤器。</li>
<li>缓存击穿：对热点 Key 增加互斥更新或使用逻辑过期 + 后台刷新。</li>
<li>缓存雪崩：过期时间随机化，设置多级限流与降级策略。</li>
</ul>
<hr>
<h2 id="服务发现与配置etcd">服务发现与配置：etcd<a aria-hidden="true" tabindex="-1" href="#服务发现与配置etcd"><span class="anchor-link"> #</span></a></h2>
<p>etcd 是一个基于 Raft 共识算法的分布式键值存储，常用于服务注册与发现、动态配置与分布式协调：</p>
<ul>
<li>服务注册与发现：
<ul>
<li>注册：实例以租约（Lease + TTL）的方式在 <code>/services/{name}/{instanceId}</code> 下写入自身地址与元信息，并定期 KeepAlive。</li>
<li>发现：网关或调用方 Watch 对应前缀变化，感知实例上下线并更新本地路由。</li>
<li>健壮性：租约到期自动删除键，避免“僵尸实例”；客户端应实现指数退避与抖动重连。</li>
</ul>
</li>
<li>动态配置：
<ul>
<li>将配置组织在 <code>/config/{app}/{env}/</code> 前缀下，发布方写入，客户端 Watch 并热更新。</li>
<li>大体量配置建议拆分模块/分段加载；敏感配置需配合加密或专用配置中心。</li>
</ul>
</li>
<li>协调能力：
<ul>
<li>分布式锁/Leader 选举：使用 etcd concurrency API（基于会话的 Mutex/Campaign）。</li>
<li>典型用途：任务分片调度、单主写入、限流配额的集中仲裁。</li>
</ul>
</li>
<li>运维要点：
<ul>
<li>集群规模以 3 或 5 节点为宜（奇数投票，过大反而降低写性能）。</li>
<li>合理设置快照与压缩（Compaction），避免 Watch Backlog 过大；对高延迟链路设置超时与心跳阈值。</li>
</ul>
</li>
</ul>
<p>在本文架构中，etcd 承担“服务注册与发现 + 动态配置 + 协调原语”的基础设施角色，辅助网关路由与内部服务寻址。</p>
<hr>
<h2 id="通信与一致性策略">通信与一致性策略<a aria-hidden="true" tabindex="-1" href="#通信与一致性策略"><span class="anchor-link"> #</span></a></h2>
<ul>
<li>通信方式：
- 同步 RPC（HTTP/gRPC）用于强实时、强一致的小闭环。
- 异步消息总线用于解耦与削峰填谷（RocketMQ/Kafka/RabbitMQ/Redis Stream 等）。</li>
<li>重试与幂等：
<ul>
<li>下游短暂不可用时可重试；所有可重试的写操作必须幂等（去重键、幂等令牌）。</li>
</ul>
</li>
<li>最终一致性：
<ul>
<li>Saga/Outbox Pattern：用本地事务 + 事件外发保证变更可达，失败则编排补偿。</li>
</ul>
</li>
<li>全局标识：
<ul>
<li>雪花算法/号段缓存生成可排序的分布式 ID，便于路由与审计。</li>
</ul>
</li>
</ul>
<hr>
<h2 id="消息中间件rocketmqkafka">消息中间件：RocketMQ/Kafka<a aria-hidden="true" tabindex="-1" href="#消息中间件rocketmqkafka"><span class="anchor-link"> #</span></a></h2>
<p>RocketMQ/Kafka这类是高性能、低延迟的分布式消息中间件，具有完备的消息语义与企业级特性：</p>
<ul>
<li>核心概念：Topic、Tag、Message Key、Producer Group、Consumer Group。</li>
<li>投递语义：默认至少一次（At-Least-Once），精确一次需在业务层通过幂等键/去重表实现。</li>
<li>顺序消息：基于分片键（Sharding Key）在队列内保证局部有序，适用于订单状态流、资金流水等场景。</li>
<li>事务消息：
<ul>
<li>流程：发送半消息（Prepare）→ 执行本地事务 → Commit/Rollback；</li>
<li>Broker 可回查事务状态（Transaction Check）以防悬挂，生产者需实现回查接口；</li>
<li>常用于 Saga/Outbox 的可靠事件外发。</li>
</ul>
</li>
<li>延迟/定时消息：内建多级延迟；配合重试策略实现柔性事务与削峰。</li>
<li>重试与死信：
<ul>
<li>消费失败按重试策略回投，超过阈值进入 DLQ（死信队列），便于排查与人工补偿。</li>
</ul>
</li>
<li>消费模式：集群（负载均衡）与广播（全部消费）；通常业务消费采用集群模式。</li>
<li>设计建议：
<ul>
<li>消息幂等：使用业务唯一键（如订单号）去重；消费者侧“先判重后执行”。</li>
<li>负载与扩展：按吞吐与有序需求规划队列数；避免消息体过大（建议 &#x3C; 1MB），大对象走存储，消息只放引用。</li>
<li>可观测：开启消息轨迹与指标上报，关注堆积、耗时与回查比例。</li>
</ul>
</li>
</ul>
<p>在架构中，RocketMQ/Kafka 作为“事件总线”，支撑跨服务解耦、削峰填谷与最终一致性：</p>
<ul>
<li>Outbox：业务表 + outbox 表本地事务提交，后台发布到 RocketMQ，消费侧幂等入库。</li>
<li>Saga：编排器基于事件推进/回滚步骤，失败走补偿消息。</li>
</ul>

<h2 id="拓扑与部署">拓扑与部署<a aria-hidden="true" tabindex="-1" href="#拓扑与部署"><span class="anchor-link"> #</span></a></h2>
<p>典型拓扑：</p>
<ul>
<li>网关（Gateway）：对外唯一入口，做鉴权、路由、限流与聚合。</li>
<li>服务注册与发现：基于 etcd 的注册表与 Watch，实现动态寻址与健康检查（也可配合 Sidecar）。</li>
<li>内部服务：按域划分的微服务/Actor 群组，互相通过 RPC/消息交互。</li>
<li>数据与缓存层：MongoDB 集群 + Redis 集群，结合副本集、哨兵与分片设计。</li>
<li>消息层：RocketMQ Broker 集群 + NameServer，按 Topic/队列扩展吞吐。</li>
</ul>
